*** What is this project about?
The project is to build a recommendations engine for the recipes to cook based on what ingredients we have

*** What needs to be build (MVP state Requirements)
We can start this iteratively, but hte core components include

1. [x] Pantry ingredient entry
  1.1 [x] users can submit the ingredients and quantities they have, like 3 tomatoes, 5 potatoes, 1 cheese etc.
  1.2 [x] users can start typing an ingredient and see matching autocomplete suggestions.
  1.3 [x] once users pick an ingredient, the app asks how much they currently have available.
  1.4 [x] users can keep adding ingredients until they hit the "finish" action.
  1.5 [x] users can add any ingredient they have, even when it is not already present in our ingredient list.
  1.6 [x] new ingredients are added to our ingredient list/catalog so they can be suggested in the future.

2. [x] Recipe recommendation engine
  2.1 [x] once all ingredients are submitted, the app recommends dishes/recipes the user can cook with what they have.
  2.2 [x] the recommendation engine first searches our existing recipe book/database for matching recipes.
  2.3 [x] a recipe is considered a useful match only if at least 60% of its required ingredients are available in the user's pantry.
  2.4 [x] every returned recipe must satisfy the 60% match rule.
  2.5 [x] the recommender normalizes common ingredient variations, such as singular/plural names.
  2.6 [x] the recommender does not penalize users for missing basic pantry staples like salt, pepper, oil, and water.
  2.7 [x] users can optionally select a cuisine type for recommendations.
  2.8 [x] if the user selects a cuisine, that cuisine is treated as a hard filter.
  2.9 [x] all returned recipes must match the selected cuisine and still satisfy the 60% pantry match rule.
  2.10 [x] a recipe is not recommended if its core ingredient is missing, even when the rest of the supporting ingredients satisfy the 60% match rule.
  2.11 [x] each recipe includes a core_ingredients list so the recommender can validate the ingredients that define the dish.
  2.12 [x] inference from recipe titles is only a fallback for core ingredients.
  2.13 [x] recommendations are ranked by match quality, cooking time, and stable recipe name ordering.
  2.14 [x] the recommendation response should always try to return 5 recipes.
  2.15 [ ] users should be able to select a dietary preference such as vegetarian, non-vegetarian, or vegan.
  2.16 [ ] dietary preference should filter recommendations alongside pantry match and cuisine preference.
  2.17 [ ] in the future, the recommendation count should be configurable from the user prompt or user preferences.

3. [x] Recipe discovery and backfill
  3.1 [x] the 5 recommendations can be a mix of recipes from our own recipe book and web-discovered recipes.
  3.2 [x] if our recipe book does not have enough useful matches to fill 5 recommendations, the app searches trusted recipe sources for the closest matching recipes.
  3.3 [x] when we find a useful recipe from the web, we store it in our recipe book so the same or similar future request can be served from our own data first.
  3.4 [x] for the MVP web discovery implementation, we start with a public recipe API such as TheMealDB before expanding to broader website scraping.
  3.5 [x] if our local recipe book and TheMealDB cannot fill 5 useful matches, the backend uses curated cuisine-specific websites as fallbacks by reading structured recipe metadata from recipe pages.
  3.6 [x] website discovery is source-specific and controlled rather than an unrestricted crawl of the public internet.
  3.7 [x] initial curated website sources include Indian Healthy Recipes for Indian recipes, Omnivore's Cookbook for Chinese recipes, Vincenzo's Plate for Italian recipes, and Mexican Please plus Simply Recipes for Mexican recipes.

4. [ ] Recipe upload and import
  4.1 [ ] users should be able to upload recipe images or files so the app can extract recipe details and add them to our recipe book.
  4.2 [ ] supported uploads should include recipe card photos, screenshots, and PDFs, including recipe cards from services like EveryPlate.
  4.3 [ ] uploaded recipe extraction should use OCR/computer vision where needed.
  4.4 [ ] uploaded recipe extraction should identify the recipe name, ingredients, quantities, instructions, cuisine when available, cooking time when available, source/notes, and core_ingredients.
  4.5 [ ] uploaded recipe extraction should include a user review/confirmation step before saving the parsed recipe into our recipe book.

5. [x] Recipe links
  5.1 [x] users can open the desired recipe online if a source URL is already present.

*** Future Scope
1. [ ] User recipe and pantry management
  1.1 [ ] users can add their own recipes which we store in our central recipes DB.
  1.2 [ ] users can add the groceries they bought on shopping day, so we know what they currently have in their kitchen.
  1.3 [ ] have our own DB collection of all recipes, including recipes we scrape from the web.

2. [ ] Personalization and notifications
  2.1 [ ] make this into an app that studies users' cooking and eating preferences.
  2.2 [ ] send notifications at appropriate times of day suggesting a particular recipe.

3. [ ] Social and marketplace features
  3.1 [ ] users can share recipes or take pictures of their food and share them on the internet.
  3.2 [ ] users can create profiles for catering businesses.
  3.3 [ ] we can act as the online marketplace for such services.

4. [x] Cuisine preferences
  4.1 [x] add a new button/control to let users pick cuisines such as Indian, Italian, Mexican, and Chinese.

*** Technologies to use

  1.1 [x] backend uses python or java.

2. [x] Web server
  2.1 [x] webserver can use FastAPI or flask as they are compatible with python.

3. [x] Data storage
  3.1 [x] use a KV-style store to store each recipe and its list of ingredients.
  3.2 [ ] for future relational entities, we can use simple postgres or any other relational DB.

4. [x] Frontend
  4.1 [x] use a lightweight JavaScript frontend for browser interactions.
  4.2 [x] keep recommendation, ingredient catalog, recipe storage, and future web discovery logic in the Python backend.

*** What is this project about?
The project is to build a recommendations engine for the recipes to cook based on what ingredients we have

*** What needs to be build (MVP state Requirements)
We can start this iteratively, but hte core components include
1. A web interface where users submit the form of the ingredients and the quantities they have (like for example, 3 tomatoes, 5 potatoes, 1 cheese etc). 
1.1 we can allow users to start typing the ingredient (they start typing tom and we should populate all possible options nad they will chose/select the appropriate one)
1.2 once they pick the ingredients, we must prompt to ask how many they currently have available.
1.3 at every step we can ask them if that is all they have and once they hit "finish" action, we can say that its all the set of ingredients they have.
1.4 users must be able to add any ingredient they have, even when it is not already present in our ingredient list.
1.5 when a new ingredient is submitted, we must add it to our ingredient list/catalog so it can be suggested in the future.

2. Once all the ingredients are submitted, we must recommend top5 dishes/recipes they can cook with what they have, and give them the list ranked by cooking time.
2.1 the recommendation engine must first search our existing recipe book/database for matching recipes.
2.2 a recipe is considered a useful match only if at least 60% of its required ingredients are available in the user's pantry.
2.3 the recommendation response should always try to return 5 recipes.
2.4 the 5 recommendations can be a mix of recipes from our own recipe book and web-discovered recipes, but every returned recipe must satisfy the 60% match rule.
2.5 if our recipe book does not have enough useful matches to fill 5 recommendations, the app should search/scrape the web for the closest matching recipes.
2.6 when we find a useful recipe from the web, we must store it in our recipe book so the same or similar future request can be served from our own data first.
2.7 in the future, the recommendation count should be configurable from the user prompt or user preferences.
2.8 for the MVP web discovery implementation, we can start with a public recipe API such as TheMealDB before expanding to broader website scraping.
2.9 the recommender should normalize common ingredient variations, such as singular/plural names, and should not penalize users for missing basic pantry staples like salt, pepper, oil, and water.

3. We can link them the desired recipes online if they are already present.

*** Future Scope
1. Users can add their own recipes which we store in our central recipes DB
2. Users can add all the list of groceries they bough on their shopping day, so that we have an idea of what exactly they currently have in their kitchen.
3. Based on #2 We want to make this into an app, such that we study users cooking and eating preferences and send notifications at appropriate time of day suggesting a particular recipe is good
4. users can share the recipes or take pics of their food and share it in the internet.
5. users can create their profiles such that they can use it as their catering business, we can act as the online marketplace for such services.

*** technologies to use
1. backend : python or java
2. webserver : can use FastAPI or flask as they are compatible with python
3. DB : we need a KV store to store each recipe and the list of ingredients but for other relational entities, we can use a simple postgres or any other realational db
4. frontend : use a lightweight JavaScript frontend for browser interactions, while keeping recommendation, ingredient catalog, recipe storage, and future web discovery logic in the Python backend
